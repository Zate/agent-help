Frequently Asked Questions

agent-help is MCP-lite for CLIs

Two flags: --agent-help for compact command discovery and --agent-out for dense structured results.

What is agent-help?

agent-help is MCP-lite for command-line tools. Instead of building an MCP server, plugin, or skill just so an AI agent can use your CLI, you add --agent-help and --agent-out.

Humans still use --help. Scripts still use --json. Agents get a tiny native surface that is easy to discover and cheap to keep in context.

What problem does it solve?

Agents are surprisingly bad at using ordinary CLIs from human help text. They parse prose, guess required flags, miss enum values, and waste context on decoration.

ah1 tool :: example data CLI
cmd search <query> [--limit int] :: search records
cmd show <id> :: display one record
more? tool <cmd> --agent-help

No prose. No guessing. No separate docs at runtime.

Is this a replacement for MCP?

No. MCP is great when you need a real tool server, resources, long-lived sessions, auth mediation, or a protocol boundary.

agent-help is for the smaller case: “I already have a CLI; I want agents to use it reliably.” You can still build an MCP server on top later.

Is this a replacement for skills?

No. Skills are useful for teaching an agent a workflow or domain. But an agent should not need a skill just to learn how to invoke tool search <query> --limit 5.

This repository includes a SKILL.md, but it is only a build-time helper for coding agents implementing agent-help in a CLI.

Why not just use --help?

Human help is optimized for people: prose, sections, alignment, color, aliases, and examples. Agents can read it, but they waste tokens and still infer structure.

cmd deploy <env> [--dry-run bool] :: deploy service

That one line tells an agent the command, required argument, useful optional flag, and purpose.

Why not just use JSON?

Keep --json for software. agent-help does not replace it.

JSON is verbose for LLM context and does not solve command discovery, pagination, recovery hints, or exact next commands.

ok issues count=2 more=1
issues[#2]{id,status,title}:
  BUG-1,open,login fails
  BUG-2,triage,slow search
next page "tool issues --cursor abc --agent-out"

What does an agent need before using an agent-help CLI?

Nothing special. The expected path is: run tool --help, see the breadcrumb, run tool --agent-help, and follow the records from there.

If you have to give the agent this website, the spec, llms-full.txt, or a skill before it can use the CLI, the implementation has missed the point.

Then why does this repo have a spec and llms-full.txt?

Because implementers need guidance. Those files teach a human or coding agent how to add the two flags to a CLI and keep output consistent. They are not runtime documentation for agents using your CLI.

How much do I need to implement?

  1. Add the breadcrumb to --help.
  2. Add tool --agent-help for a command index.
  3. Add tool subcmd --agent-help for command detail.
  4. Add --agent-out only where structured results matter.

You can stop after discovery and still get value. See the implementation guides for framework wiring.
